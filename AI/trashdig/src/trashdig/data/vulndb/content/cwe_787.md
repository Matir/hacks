# CWE-787: Out-of-bounds Write

### Description
An Out-of-bounds Write occurs when the software writes data past the end, or before the beginning, of the intended buffer. This typically happens due to incorrect calculation of buffer sizes, improper loop boundaries, or failure to validate the length of input data before copying it into a fixed-size memory allocation.

In languages like C and C++, which do not provide automatic memory management or bounds checking, the responsibility falls entirely on the developer. When a program writes outside the bounds of an allocated memory block, it can corrupt adjacent data, including control flow information such as function pointers or return addresses.

### Impact
The impact of an out-of-bounds write is usually severe and can include:
*   **Arbitrary Code Execution (ACE):** By overwriting return addresses on the stack or function pointers in the heap, an attacker can redirect the program's execution flow to run malicious code.
*   **Denial of Service (DoS):** Overwriting critical memory structures often leads to application crashes or system instability.
*   **Data Corruption:** The vulnerability can be used to modify sensitive variables (e.g., a `is_admin` flag) to bypass security checks.

### Remediation
To mitigate CWE-787, developers should implement the following strategies:

1.  **Use Safe Functions:** Replace dangerous functions like `strcpy`, `sprintf`, and `gets` with safer alternatives that require a maximum length (e.g., `strncpy`, `snprintf`, or `fgets`). Note that `strncpy` may not null-terminate the string if the source is longer than the limit, so manual termination is required.
2.  **Input Validation:** Always validate the length of input data before performing memory operations. Ensure that the size of the source data does not exceed the capacity of the destination buffer.
3.  **Bounds Checking:** When using loops to populate arrays or buffers, ensure the index is always within the valid range (0 to N-1).
4.  **Modern Language Features:** Use higher-level abstractions that handle memory safety automatically, such as `std::string` or `std::vector` in C++, or migrate to memory-safe languages like Rust or Go.
5.  **Compiler Protections:** Enable security features such as Address Space Layout Randomization (ASLR), Data Execution Prevention (DEP/NX), and Stack Canaries (e.g., `-fstack-protector-all` in GCC).

### Examples

#### Vulnerable Code (C)
In this example, the program accepts input from the user and copies it into a fixed-size buffer using `strcpy` without checking the length.

```c
#include <stdio.h>
#include <string.h>

void process_data(char *input) {
    char buffer[16];
    // VULNERABLE: strcpy does not check the size of input
    // If input is longer than 15 characters + null terminator, 
    // it will write past the end of 'buffer'.
    strcpy(buffer, input);
    printf("Buffer content: %s\n", buffer);
}

int main(int argc, char **argv) {
    if (argc > 1) {
        process_data(argv[1]);
    }
    return 0;
}
```

#### Secure Code (C)
This version uses `strncpy` and ensures the buffer is correctly terminated, preventing the out-of-bounds write.

```c
#include <stdio.h>
#include <string.h>

void process_data(char *input) {
    char buffer[16];
    
    // SECURE: Limit the copy to the size of the destination buffer
    strncpy(buffer, input, sizeof(buffer) - 1);
    
    // Ensure the string is null-terminated
    buffer[sizeof(buffer) - 1] = '\0';
    
    printf("Buffer content: %s\n", buffer);
}

int main(int argc, char **argv) {
    if (argc > 1) {
        process_data(argv[1]);
    }
    return 0;
}
```
