### Description

CWE-770 occurs when an application allocates resources—such as memory, disk space, CPU time, network sockets, or file handles—based on untrusted input without performing adequate validation of the requested amount or enforcing global limits. This vulnerability is a fundamental design flaw where the system assumes that resource requests will always be within reasonable bounds. Attackers can exploit this by requesting excessively large amounts of a resource, leading to system exhaustion.

While often associated with memory allocation (e.g., `malloc` or `make`), it also applies to:
*   **Disk space:** Allowing users to upload files of arbitrary size.
*   **CPU:** Operations that trigger complex computations (like regex or deep recursion) based on user input.
*   **Network:** Opening an unlimited number of concurrent connections or sessions.

### Impact

The primary impact is **Denial of Service (DoS)**. Successful exploitation can lead to:
*   **Application Crashing:** Out-of-memory (OOM) errors causing the process to terminate.
*   **System Instability:** The entire host operating system may become unresponsive if global resources (like physical RAM or swap) are exhausted.
*   **Economic Denial of Service (EDoS):** In cloud environments, auto-scaling mechanisms may trigger the allocation of more instances to handle the resource load, leading to massive, unexpected financial costs for the organization.
*   **Service Unavailability:** Legitimate users are unable to access the application because all available handles or threads are occupied by the attacker.

### Remediation

To mitigate CWE-770, developers should adopt a "defense in depth" strategy regarding resource management:

1.  **Input Validation:** Always validate that user-supplied integers used for allocation (lengths, sizes, counts) fall within a strict, predefined range.
2.  **Hard Quotas:** Implement hard limits on the maximum amount of a resource any single request or user can consume.
3.  **Throttling and Rate Limiting:** Limit the frequency of resource-intensive requests from a single IP address or user account.
4.  **Resource Pooling:** Use pools for limited resources (like database connections) rather than creating new ones on demand based on user input.
5.  **Timeouts:** Implement aggressive timeouts for resource usage to ensure that resources are eventually reclaimed.

### Examples

#### Example 1: Memory Allocation (Go)

**Vulnerable:**
In this example, the `size` parameter is taken directly from a user request and used to allocate a slice. An attacker could pass a value like `2,000,000,000` to crash the service.

```go
func handleRequest(w http.ResponseWriter, r *http.Request) {
    sizeStr := r.URL.Query().Get("size")
    size, _ := strconv.Atoi(sizeStr)
    
    // VULNERABLE: Unbounded allocation based on user input
    buffer := make([]byte, size)
    render(w, buffer)
}
```

**Secure:**
The application checks the requested size against a constant `MAX_ALLOWED_SIZE` before proceeding with the allocation.

```go
const MAX_ALLOWED_SIZE = 1024 * 1024 // 1MB limit

func handleRequest(w http.ResponseWriter, r *http.Request) {
    sizeStr := r.URL.Query().Get("size")
    size, err := strconv.Atoi(sizeStr)
    
    // SECURE: Validate input against a hard limit
    if err != nil || size < 0 || size > MAX_ALLOWED_SIZE {
        http.Error(w, "Invalid size", http.StatusBadRequest)
        return
    }
    
    buffer := make([]byte, size)
    render(w, buffer)
}
```

#### Example 2: File Reading (Python)

**Vulnerable:**
The `read()` method without a limit or with a user-controlled limit can read a massive file entirely into memory.

```python
@app.route('/preview')
def preview_file():
    byte_count = request.args.get('bytes', type=int)
    with open('large_data.log', 'rb') as f:
        # VULNERABLE: An attacker can specify a massive byte_count
        data = f.read(byte_count)
    return data
```

**Secure:**
The application enforces a maximum chunk size using `min()`.

```python
MAX_PREVIEW_SIZE = 5000

@app.route('/preview')
def preview_file():
    requested_bytes = request.args.get('bytes', type=int)
    
    # SECURE: Enforce a maximum threshold
    safe_size = min(requested_bytes or 0, MAX_PREVIEW_SIZE)
    
    if safe_size <= 0:
        return "Invalid size", 400

    with open('large_data.log', 'rb') as f:
        data = f.read(safe_size)
    return data
```
