# Description
CWE-915, commonly known as **Mass Assignment** or **Overposting**, occurs when an application takes user-supplied input and automatically binds it to internal object attributes without sufficient filtering or validation. 

Modern web frameworks often provide "auto-binding" or "object-relational mapping" (ORM) features that allow developers to easily map HTTP request parameters (from JSON, XML, or Form data) directly into programming language objects or database models. If the application does not explicitly restrict which attributes can be modified, an attacker can craft a request that includes unexpected fields. By including attributes that exist in the internal object but were not intended to be exposed (such as `is_admin`, `balance`, or `role`), the attacker can manipulate the application's state or logic.

# Impact
Exploiting CWE-915 can lead to severe security compromises, including:
*   **Privilege Escalation**: An attacker can grant themselves administrative rights by adding `"is_admin": true` to a profile update request.
*   **Account Takeover**: By modifying fields like `email` or `password_reset_token` on a user object that wasn't properly guarded.
*   **Data Tampering**: Unauthorized modification of sensitive data such as account balances, item prices in a shopping cart, or status flags (e.g., marking an order as `paid`).
*   **Security Bypass**: Overwriting internal flags used for multi-factor authentication or account verification.

# Remediation
To prevent Mass Assignment, developers should adopt a "secure by default" approach to data binding:

1.  **Use Data Transfer Objects (DTOs) or Input Models**: Instead of binding request data directly to database entities, bind them to a specific class that only contains the fields a user is allowed to provide. Map the DTO to the entity manually.
2.  **Allow-listing (Strict Filtering)**: Use framework-specific features to explicitly define which fields are allowed. 
    *   In **Ruby on Rails**, use "Strong Parameters" (`params.require(:user).permit(:name, :email)`).
    *   In **ASP.NET**, use the `[Bind]` attribute or define specific ViewModels.
3.  **Manual Assignment**: Explicitly assign values from the request to the object. While more verbose, it is the most secure method.
4.  **Ignore Sensitive Attributes**: Use framework annotations to mark sensitive fields as non-bindable (e.g., `@JsonIgnore` in Java/Jackson or `[BindNever]` in C#).

# Examples

## Vulnerable Example (Node.js/Express)
The following code is vulnerable because it uses `Object.assign` to merge the entire request body into the user object stored in the database.

```javascript
app.patch('/api/user/profile', async (req, res) => {
  const user = await User.findById(req.user.id);
  
  // VULNERABLE: Direct mass assignment from untrusted input
  Object.assign(user, req.body);
  
  await user.save();
  res.json({ message: "Profile updated successfully" });
});
// An attacker could send: PATCH {"name": "Attacker", "role": "admin"}
```

## Secure Example (Node.js/Express)
The secure version explicitly extracts only the fields that are permitted for modification.

```javascript
app.patch('/api/user/profile', async (req, res) => {
  const user = await User.findById(req.user.id);
  
  // SECURE: Explicitly defining allowed fields (Allow-listing)
  const { displayName, bio, location } = req.body;
  
  if (displayName !== undefined) user.displayName = displayName;
  if (bio !== undefined) user.bio = bio;
  if (location !== undefined) user.location = location;
  
  await user.save();
  res.json({ message: "Profile updated successfully" });
});
```

## Vulnerable Example (Ruby on Rails)
```ruby
def update
  @user = User.find(params[:id])
  # VULNERABLE: permit! allows all parameters to be updated
  @user.update(params[:user].permit!)
end
```

## Secure Example (Ruby on Rails)
```ruby
def update
  @user = User.find(params[:id])
  # SECURE: Only name and bio can be updated via this controller
  @user.update(params.require(:user).permit(:name, :bio))
end
```
