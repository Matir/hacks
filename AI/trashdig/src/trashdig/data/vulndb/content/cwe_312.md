# CWE-312: Cleartext Storage of Sensitive Information

## Description
CWE-312 occurs when an application stores sensitive information in a non-volatile location (such as a hard drive, database, or external storage) without using encryption or strong hashing. Sensitive information includes, but is not limited to, passwords, API keys, personally identifiable information (PII), credit card numbers, and session tokens.

When data is stored in cleartext, any person or process with access to the storage medium can read the information. This includes local users, attackers who have compromised the server via other means (like Path Traversal), or unauthorized personnel accessing backups and logs.

## Impact
The impact of cleartext storage is often critical:
*   **Credential Theft:** Attackers can gain administrative access to the system or other integrated services by stealing stored passwords or API keys.
*   **Privacy Violations:** Exposure of PII can lead to identity theft and severe legal repercussions.
*   **Regulatory Non-compliance:** Storing sensitive data in cleartext often violates industry standards and regulations such as GDPR, HIPAA, and PCI-DSS.
*   **Loss of Reputation:** Data breaches involving cleartext secrets significantly damage user trust.

## Remediation
To mitigate this vulnerability, follow the principle of "Defense in Depth" and ensure all sensitive data is protected at rest.

1.  **Encryption:** Use strong, industry-standard symmetric encryption algorithms (e.g., AES-256-GCM) to encrypt sensitive data before storage. Manage encryption keys securely using hardware security modules (HSM) or dedicated key management services (KMS).
2.  **Hashing:** For passwords, never use reversible encryption. Instead, use cryptographically strong, salted, and iterative one-way hashes like **Argon2**, **bcrypt**, or **scrypt**.
3.  **Platform-Specific Secure Storage:**
    *   **Android:** Use the `EncryptedSharedPreferences` or the Android Keystore system.
    *   **iOS:** Use the Keychain Services for small secrets.
    *   **Web:** Avoid storing sensitive data in `localStorage` or `sessionStorage`. Use `HttpOnly` and `Secure` cookies for session management.
4.  **Data Minimization:** Do not store sensitive information unless it is absolutely necessary for the application's functionality.

## Examples

### Vulnerable Code (Python)
In this example, user credentials are saved to a local configuration file in plain text.

```python
def save_credentials(username, password):
    # VULNERABLE: Writing sensitive password in cleartext to a file
    with open("config.txt", "w") as f:
        f.write(f"user={username}\n")
        f.write(f"pass={password}\n")

save_credentials("admin", "P@ssw0rd123!")
```

### Secure Code (Python)
In the secure version, we use a strong hashing algorithm (bcrypt) for the password and avoid storing it in a way that can be reversed.

```python
import bcrypt

def save_credentials(username, password):
    # SECURE: Hash the password using a strong algorithm with a salt
    salt = bcrypt.gensalt()
    hashed_password = bcrypt.hashpw(password.encode('utf-8'), salt)
    
    with open("config.txt", "wb") as f:
        f.write(f"user={username}\n".encode('utf-8'))
        f.write(b"pass_hash=" + hashed_password + b"\n")

save_credentials("admin", "P@ssw0rd123!")
```

### Vulnerable Code (JavaScript/Web)
Storing a JWT or sensitive token in `localStorage` makes it vulnerable to Cross-Site Scripting (XSS).

```javascript
// VULNERABLE: Sensitive token stored in cleartext accessible by any script
const token = "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...";
localStorage.setItem('auth_token', token);
```

### Secure Code (JavaScript/Web)
Sensitive tokens should be stored in cookies with security flags to prevent access from client-side scripts.

```javascript
// SECURE: Set token in a Secure, HttpOnly cookie (usually done on the server-side)
// This prevents JavaScript from accessing the token via document.cookie
document.cookie = "auth_token=eyJhbGci...; Secure; HttpOnly; SameSite=Strict";
```
