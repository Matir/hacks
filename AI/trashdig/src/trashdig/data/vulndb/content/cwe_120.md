# CWE-120: Buffer Copy without Checking Size of Input ('Classic Buffer Overflow')

### Description
CWE-120 is the most fundamental member of the buffer overflow family. It occurs when a program copies data from one location to another (usually a buffer) without verifying that the destination buffer is large enough to hold the data. In C and C++, many standard library functions do not perform bounds checking by default. When an application accepts input from an untrusted source and passes it to these functions, it can overwrite adjacent memory on the stack or heap. This can lead to the corruption of program variables, return addresses, and function pointers.

### Impact
The impact of a classic buffer overflow is often severe:
*   **Arbitrary Code Execution (ACE):** By overwriting a function's return address on the stack, an attacker can redirect the execution flow to a malicious payload (shellcode) injected into memory.
*   **Denial of Service (DoS):** Overwriting critical memory structures usually causes the application to crash.
*   **Data Corruption:** An attacker may overwrite nearby variables to change the application's logic (e.g., changing a `is_admin` boolean flag).
*   **Information Disclosure:** In some variations, overflows can be used to leak sensitive information from memory.

### Remediation
To prevent CWE-120, developers should adopt a defense-in-depth approach:

1.  **Use Bounds-Checked Functions:** Replace dangerous functions with their safer counterparts:
    *   Replace `gets()` with `fgets()`.
    *   Replace `strcpy()` with `strncpy()` (ensuring null termination) or `strlcpy()`.
    *   Replace `sprintf()` with `snprintf()`.
    *   Replace `strcat()` with `strncat()`.
2.  **Input Validation:** Always validate the length of input data before processing or copying it.
3.  **Use Modern Languages:** Where possible, use memory-safe languages like Rust, Go, Java, or Python that handle memory management automatically and prevent overflows.
4.  **Compiler and OS Protections:** Enable security features such as:
    *   **Stack Canaries:** Detects stack smashing before returning from a function.
    *   **ASLR (Address Space Layout Randomization):** Makes it harder to predict memory addresses.
    *   **DEP/NX (Data Execution Prevention):** Prevents execution of code in data segments like the stack or heap.

### Examples

#### Vulnerable Code (C)
In this example, the program uses `strcpy` to copy a command-line argument into a fixed-size buffer. If the user provides an argument longer than 63 characters, the stack will be corrupted.

```c
#include <string.h>
#include <stdio.h>

int main(int argc, char **argv) {
    char buffer[64];
    if (argc > 1) {
        // VULNERABLE: No check on the size of argv[1]
        strcpy(buffer, argv[1]);
        printf("Buffer content: %s\n", buffer);
    }
    return 0;
}
```

#### Secure Code (C)
This version uses `snprintf`, which limits the number of bytes written to the buffer, preventing an overflow.

```c
#include <string.h>
#include <stdio.h>

int main(int argc, char **argv) {
    char buffer[64];
    if (argc > 1) {
        // SECURE: snprintf limits the copy to the size of the buffer
        // It also ensures the string is null-terminated.
        snprintf(buffer, sizeof(buffer), "%s", argv[1]);
        printf("Buffer content: %s\n", buffer);
    }
    return 0;
}
```
