# CWE-862: Missing Authorization

## Description
CWE-862 occurs when a software application does not perform an authorization check when an actor attempts to access a resource or perform an action. While **Authentication** verifies "who a user is," **Authorization** determines "what a user is allowed to do."

When authorization is missing, an application might allow an authenticated user (or even an unauthenticated one) to access sensitive data, modify configurations, or perform administrative tasks that should be restricted to specific roles or ownership. This is often a result of developers assuming that if a user knows a specific URL or ID, they must be authorized to access it, or failing to apply security filters to new API endpoints.

## Impact
The impact of missing authorization is often severe and includes:
*   **Information Disclosure:** Unauthorized access to sensitive data (PII, financial records, intellectual property).
*   **Data Corruption:** Unauthorized modification or deletion of data belonging to other users or the system.
*   **Privilege Escalation:** A standard user gaining the ability to perform administrative functions.
*   **Account Takeover:** Modifying credentials or security settings of other users.

## Remediation

### General Principles
1.  **Deny by Default:** Design the system to restrict access unless explicitly allowed.
2.  **Centralized Authorization:** Use a single, well-vetted library or framework for authorization rather than implementing ad-hoc checks in every function.
3.  **Check at the Functional Level:** Every request that accesses sensitive data or performs a state change must be checked against the user's permissions.
4.  **Enforce Ownership:** Ensure that users can only access resources they own (Insecure Direct Object Reference or IDOR mitigation).

### Language-Specific Recommendations
*   **Java (Spring Security):** Use annotations like `@PreAuthorize`, `@PostAuthorize`, or `@Secured` on controller methods or service layers. Ensure `HttpSecurity` configurations require authentication and specific roles for matching paths.
*   **Python (Django/Flask):** Use decorators such as `@login_required` and `@permission_required` (Django) or custom wrappers (Flask). In Django, utilize the built-in Permission system and `UserPassesTestMixin` for class-based views.
*   **Node.js (Express):** Implement middleware that checks JWT roles or session permissions before reaching the route handler.

## Examples

### Vulnerable Example (Python/Flask)
In this example, the application deletes a user based on the `user_id` provided in the URL. It checks if the requester is logged in, but fails to check if the requester has administrative privileges.

```python
@app.route('/api/v1/users/delete/<int:user_id>', methods=['POST'])
def delete_user(user_id):
    # AUTHENTICATION check only
    if 'user' not in session:
        abort(401)
    
    # MISSING AUTHORIZATION: 
    # Any logged-in user can delete any other user!
    db.delete_user(user_id)
    return jsonify({"status": "success"}), 200
```

### Secure Example (Python/Flask)
In the secure version, we implement a check to ensure the current session user has the 'admin' role before proceeding with the deletion.

```python
from functools import wraps

def admin_required(f):
    @wraps(f)
    def decorated_function(*args, **kwargs):
        if session.get('role') != 'admin':
            abort(403) # Forbidden
        return f(*args, **kwargs)
    return decorated_function

@app.route('/api/v1/users/delete/<int:user_id>', methods=['POST'])
@admin_required # Authorization check
def delete_user(user_id):
    if 'user' not in session:
        abort(401)
        
    db.delete_user(user_id)
    return jsonify({"status": "success"}), 200
```

### Vulnerable Example (Java/Spring Boot)
The method below is accessible to any authenticated user because it lacks a specific security annotation.

```java
@RestController
@RequestMapping("/api/orders")
public class OrderController {

    @GetMapping("/{orderId}")
    public Order getOrder(@PathVariable Long orderId) {
        // Vulnerable: Any authenticated user can view any order by ID
        return orderRepository.findById(orderId);
    }
}
```

### Secure Example (Java/Spring Boot)
The secure version ensures that the user requesting the order is either an admin or the owner of the order.

```java
@RestController
@RequestMapping("/api/orders")
public class OrderController {

    @GetMapping("/{orderId}")
    @PreAuthorize("hasRole('ADMIN') or #order.owner == authentication.name")
    public Order getOrder(@PathVariable Long orderId) {
        return orderRepository.findById(orderId);
    }
}
```
