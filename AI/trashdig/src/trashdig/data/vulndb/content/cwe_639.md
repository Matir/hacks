# CWE-639: Authorization Bypass Through User-Controlled Key

### Description
CWE-639, often referred to as **Insecure Direct Object Reference (IDOR)**, occurs when an application provides direct access to objects based on user-supplied input. In this scenario, the application uses a user-controlled key (such as a database ID, a filename, or a primary key) to retrieve a record or perform an action without verifying that the authenticated user has the appropriate permissions to access that specific object.

While the user may be authenticated to the system, the application fails to perform the secondary authorization check: *"Is this user allowed to access THIS specific resource?"* Attackers exploit this by manipulating the identifier (e.g., changing `?user_id=123` to `?user_id=124`) to access data belonging to other users.

### Impact
The consequences of CWE-639 are typically severe and include:
*   **Information Exposure**: Unauthorized access to sensitive data such as Personal Identifiable Information (PII), medical records, or financial statements.
*   **Data Manipulation**: Attackers may be able to modify or delete records belonging to other users if the vulnerability exists in `POST`, `PUT`, or `DELETE` endpoints.
*   **Account Takeover**: If the vulnerable key is used in password reset or profile update flows, an attacker might modify other users' account credentials.
*   **Horizontal Privilege Escalation**: A user accessing resources of another user with the same privilege level.

### Remediation

To prevent authorization bypass through user-controlled keys, implement the following strategies:

1.  **Enforce Object-Level Access Control**: For every request that accesses data, the application must verify that the currently logged-in user owns the resource or has an explicit permission to access it.
2.  **Use Indirect Reference Maps**: Instead of exposing database IDs directly, use a temporary, session-specific mapping (e.g., User 1 sees their files as 'A', 'B', 'C' which map internally to '105', '202', '305').
3.  **Use Non-Predictable Identifiers (UUIDs)**: While not a complete fix, using version 4 UUIDs makes it computationally difficult for an attacker to guess other valid IDs compared to sequential integers.
4.  **Database-Level Scoping**: Include the user's unique identifier (from the secure session) in the database query itself (e.g., `SELECT * FROM orders WHERE id = ? AND user_id = ?`).

### Examples

#### Vulnerable Example (Python/Flask)
In this example, the application takes a `transaction_id` from the URL and fetches the transaction without checking if it belongs to the logged-in user.

```python
@app.route('/view_transaction/<transaction_id>')
@login_required
def view_transaction(transaction_id):
    # VULNERABLE: Only checks if the user is logged in, 
    # not if they own the transaction_id.
    transaction = Transaction.query.get(transaction_id)
    return render_template('transaction.html', data=transaction)
```

#### Secure Example (Python/Flask)
In the secure version, the query is scoped to the `user_id` stored in the secure, server-side session.

```python
@app.route('/view_transaction/<transaction_id>')
@login_required
def view_transaction(transaction_id):
    # SECURE: Validates ownership by including the user_id in the query
    transaction = Transaction.query.filter_by(
        id=transaction_id, 
        user_id=session['user_id']
    ).first()
    
    if not transaction:
        abort(404) # Or 403 Forbidden
        
    return render_template('transaction.html', data=transaction)
```

#### Vulnerable Example (Node.js/Express)
```javascript
app.get('/api/user/profile/:id', (req, res) => {
  // VULNERABLE: Uses the ID from the URL directly to find a user document
  db.collection('users').findOne({ _id: req.params.id }, (err, user) => {
    res.json(user);
  });
});
```

#### Secure Example (Node.js/Express)
```javascript
app.get('/api/user/profile/:id', (req, res) => {
  const authenticatedUserId = req.session.userId;
  
  // SECURE: Ensures the requested ID matches the authenticated user's ID
  if (req.params.id !== authenticatedUserId) {
    return res.status(403).send('Unauthorized');
  }

  db.collection('users').findOne({ _id: req.params.id }, (err, user) => {
    res.json(user);
  });
});
```
