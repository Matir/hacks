# CWE-121: Stack-based Buffer Overflow

## Description

A stack-based buffer overflow occurs when a program writes more data to a buffer located on the stack than that buffer is allocated to hold. In languages like C and C++, memory management is often manual, and many standard library functions do not automatically check bounds. 

When a buffer is overflown on the stack, the extra data spills over into adjacent memory locations. This is particularly dangerous because the stack contains critical control information, such as the **Return Address** (the instruction pointer the CPU should jump to after the current function finishes). By carefully crafting the input, an attacker can overwrite this return address to point to malicious code (shellcode) or to existing library functions (Return-to-libc attacks), thereby hijacking the execution flow of the application.

## Impact

The impact of a stack-based buffer overflow is typically severe:

*   **Arbitrary Code Execution (ACE/RCE):** The most critical impact. An attacker can execute arbitrary commands with the privileges of the running process.
*   **Denial of Service (DoS):** Overwriting stack metadata usually leads to memory corruption that crashes the application.
*   **Privilege Escalation:** If the vulnerable application runs with elevated privileges (e.g., setuid root), a successful exploit grants the attacker those same privileges.
*   **Information Disclosure:** In some variations, an attacker can read adjacent stack memory, potentially leaking sensitive data like pointers, stack canaries, or cryptographic keys.

## Remediation

### 1. Use Safe Functions
Avoid functions that do not perform bounds checking. Replace them with their safer counterparts:
*   Replace `strcpy()` with `strncpy()` or `strlcpy()`.
*   Replace `strcat()` with `strncat()`.
*   Replace `gets()` with `fgets()`.
*   Replace `sprintf()` with `snprintf()`.

### 2. Implement Input Validation
Always validate the length of input data before processing it. Ensure that the size of the source data does not exceed the size of the destination buffer.

### 3. Use Compiler and OS Protections
Modern systems provide several layers of defense-in-depth:
*   **Stack Canaries (Stack Smashing Protector):** The compiler places a random "canary" value before the return address. If the canary is modified (overwritten), the program terminates before executing the corrupted return address.
*   **Address Space Layout Randomization (ASLR):** Randomizes the memory addresses used by the stack, heap, and libraries, making it harder for attackers to predict where their code or gadgets are located.
*   **NX/DEP (No-Execute / Data Execution Prevention):** Marks the stack as non-executable, preventing the execution of code injected into a buffer.

### 4. Use Memory-Safe Languages
If possible, use higher-level languages (e.g., Rust, Go, Python, Java) that provide automatic memory management and bounds checking by default.

## Examples

### Vulnerable Code (C)
In this example, the `strcpy` function does not check the size of the input `argv[1]`. If a user provides a string longer than 63 characters, the stack will be corrupted.

```c
#include <string.h>
#include <stdio.h>

void greet_user(char *name) {
    char buffer[64];
    // VULNERABLE: strcpy does not check if 'name' fits in 'buffer'
    strcpy(buffer, name); 
    printf("Hello, %s!\n", buffer);
}

int main(int argc, char **argv) {
    if (argc > 1) {
        greet_user(argv[1]);
    }
    return 0;
}
```

### Secure Code (C)
In the secure version, `strncpy` is used to limit the number of bytes copied, and the buffer is explicitly null-terminated.

```c
#include <string.h>
#include <stdio.h>

void greet_user(char *name) {
    char buffer[64];
    // SECURE: Limit the copy to the size of the buffer minus one for the null terminator
    strncpy(buffer, name, sizeof(buffer) - 1);
    buffer[sizeof(buffer) - 1] = '\0'; 
    
    printf("Hello, %s!\n", buffer);
}

int main(int argc, char **argv) {
    if (argc > 1) {
        greet_user(argv[1]);
    }
    return 0;
}
```

### Secure Code (C++)
Using modern C++ features like `std::string` or `std::vector` eliminates the risk of manual buffer management.

```cpp
#include <iostream>
#include <string>

void greet_user(const std::string& name) {
    // SECURE: std::string handles memory management and sizing automatically
    std::string greeting = "Hello, " + name + "!";
    std::cout << greeting << std::endl;
}

int main(int argc, char **argv) {
    if (argc > 1) {
        greet_user(argv[1]);
    }
    return 0;
}
```
