# CWE-117: Improper Output Neutralization for Logs

## Description
Improper Output Neutralization for Logs, commonly known as **Log Injection** or **Log Forging**, occurs when an application includes unvalidated or unsanitized user input in its log files. Because log files are often used for security auditing, troubleshooting, and forensics, the ability of an attacker to corrupt these files is a significant security concern.

The most common technique involves injecting Carriage Return (CR) and Line Feed (LF) characters (e.g., `\r\n` or `%0D%0A`). These characters allow an attacker to terminate the current log entry and start a new, completely fabricated entry on a new line. This can be used to hide malicious activity, spoof system events, or mislead administrators during an investigation.

Additionally, if logs are viewed through a web-based dashboard (like Kibana, Splunk, or a custom admin panel) without proper output encoding, Log Injection can escalate into **Stored Cross-Site Scripting (XSS)**.

## Impact
*   **Log Forging:** Attackers can inject fake log entries to frame other users, hide their own tracks, or trick automated log analysis tools into triggering false alerts or ignoring real ones.
*   **Audit Circumvention:** By manipulating logs, an attacker can bypass compliance requirements and security auditing processes.
*   **Cross-Site Scripting (XSS):** If log viewing tools do not escape HTML/JavaScript, an attacker can inject malicious scripts that execute when an administrator views the logs.
*   **Log Corruption/Denial of Service:** Injecting massive amounts of data or binary characters can corrupt log files or exhaust disk space, making the monitoring system unavailable.

## Remediation
To prevent log injection, all data originating from untrusted sources must be neutralized before being logged.

1.  **Input Sanitization:** Strip or replace control characters, specifically Carriage Return (`\r` / `U+000D`) and Line Feed (`\n` / `U+000A`), from any data before passing it to a logging function.
2.  **Allow-listing:** Only allow known safe characters (e.g., alphanumeric) in log messages.
3.  **Use Structured Logging:** Use logging formats like JSON or libraries that treat data as parameters rather than raw strings. While this doesn't always prevent the injection into the raw file, it makes it much harder to "break out" of the log entry format.
4.  **Context-Aware Encoding:** If logs are viewed in a web browser, ensure the viewing application applies proper HTML entity encoding to the log data.

## Examples

### Vulnerable Code (Python)
In this example, an attacker can provide a username containing CRLF characters to forge a new log entry.

```python
import logging
from flask import request

@app.route('/login')
def login():
    username = request.args.get('username')
    # VULNERABLE: Direct concatenation of user input into logs
    logging.info(f"Failed login attempt for user: {username}")
    return "Login Failed", 401
```
*Attacker Input:* `admin\n[INFO] User admin logged in successfully`
*Resulting Log:*
```text
INFO:root:Failed login attempt for user: admin
[INFO] User admin logged in successfully
```

### Secure Code (Python)
The secure version sanitizes the input by replacing newline characters.

```python
import logging
from flask import request

def sanitize_log(data):
    return data.replace('\n', '_').replace('\r', '_')

@app.route('/login')
def login():
    username = request.args.get('username')
    # SECURE: Neutralize CRLF characters
    safe_username = sanitize_log(username)
    logging.info(f"Failed login attempt for user: {safe_username}")
    return "Login Failed", 401
```

### Vulnerable Code (Java)
```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public void doLogin(String username) {
    Logger logger = LoggerFactory.getLogger(getClass());
    // VULNERABLE: User-controlled string is logged directly
    logger.info("Login failed for: " + username);
}
```

### Secure Code (Java)
Using a regex to strip control characters or using a custom wrapper.
```java
public void doLogin(String username) {
    Logger logger = LoggerFactory.getLogger(getClass());
    // SECURE: Sanitize input by removing control characters
    String safeUser = username.replaceAll("[\\u0000-\\u001f]", "_");
    logger.info("Login failed for: {}", safeUser);
}
```
