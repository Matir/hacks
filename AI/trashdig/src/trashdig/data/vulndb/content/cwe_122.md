# Description
A heap-based buffer overflow is a memory corruption vulnerability that occurs when a program writes more data to a buffer allocated on the heap than the buffer can hold. The heap is a region of memory used for dynamic memory allocation (e.g., using `malloc()`, `calloc()`, or the `new` operator in C++).

Unlike stack-based overflows, which overwrite return addresses on the call stack, heap overflows overwrite metadata used by the memory allocator (chunk headers) or adjacent data objects stored on the heap. This can lead to the corruption of program logic, function pointers, or the heap's internal structures, which attackers can leverage to gain control of the execution flow.

# Impact
The impact of a heap-based buffer overflow is often severe:
*   **Arbitrary Code Execution (RCE):** By overwriting function pointers or exploiting heap management structures (like `dlmalloc` or `ptmalloc` bins), an attacker can redirect execution to malicious code.
*   **Denial of Service (DoS):** Corrupting heap metadata typically leads to a segmentation fault or an immediate crash when the memory allocator attempts to perform `free()` or `realloc()` on the corrupted area.
*   **Data Corruption:** Attackers may overwrite adjacent variables or objects to bypass authentication, change user privileges, or alter application logic.

# Remediation
To prevent heap-based buffer overflows, developers should adopt the following practices:

1.  **Use Safe String/Memory Functions:** Replace unsafe functions like `strcpy()`, `sprintf()`, and `gets()` with their safer counterparts that require a buffer size limit, such as `strncpy()`, `snprintf()`, or `fgets()`.
2.  **Validate Input Length:** Always check the length of input data before copying it into a fixed-size heap buffer. Ensure the allocation size accounts for null terminators.
3.  **Use Modern C++ Containers:** In C++, avoid manual memory management. Use `std::string`, `std::vector`, and smart pointers (`std::unique_ptr`, `std::shared_ptr`), which handle bounds checking or automatic resizing.
4.  **Adopt Memory-Safe Languages:** If possible, use languages with built-in memory safety (e.g., Rust, Go, Java, Python) that prevent buffer overflows by design.
5.  **Compiler and OS Mitigations:** Enable protections such as Address Space Layout Randomization (ASLR), Data Execution Prevention (DEP/NX), and heap integrity checks (e.g., Windows Segment Heap protections or Glibc's `MALLOC_CHECK_`).

# Examples

### Vulnerable Code (C)
In this example, the application allocates a fixed amount of memory on the heap but uses `strcpy()` to copy user input of an arbitrary length into that buffer.

```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

int main(int argc, char *argv[]) {
    if (argc < 2) return 1;

    // Allocate 64 bytes on the heap
    char *heap_buffer = (char *)malloc(64);
    if (heap_buffer == NULL) return 1;

    // VULNERABLE: strcpy does not check the length of argv[1].
    // If argv[1] > 63 characters, it overflows into adjacent heap memory.
    strcpy(heap_buffer, argv[1]);

    printf("Buffer content: %s\n", heap_buffer);

    free(heap_buffer);
    return 0;
}
```

### Secure Code (C)
The secure version uses `strncpy()` and ensures the buffer is null-terminated, preventing memory from being overwritten beyond its allocation.

```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

int main(int argc, char *argv[]) {
    if (argc < 2) return 1;

    size_t buffer_size = 64;
    char *heap_buffer = (char *)malloc(buffer_size);
    if (heap_buffer == NULL) return 1;

    // SECURE: Use strncpy to limit the number of bytes copied.
    // Ensure we leave space for the null terminator.
    strncpy(heap_buffer, argv[1], buffer_size - 1);
    heap_buffer[buffer_size - 1] = '\0'; // Explicit null termination

    printf("Buffer content: %s\n", heap_buffer);

    free(heap_buffer);
    return 0;
}
```
