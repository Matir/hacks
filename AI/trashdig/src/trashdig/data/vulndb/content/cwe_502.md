## Description

**CWE-502: Deserialization of Untrusted Data** occurs when an application receives a serialized object from an untrusted source and deserializes it without sufficient validation. 

Serialization is the process of converting complex data structures (like objects in Java, Python, or PHP) into a binary or textual format for storage or transmission. Deserialization is the reverse process. The danger arises because many serialization frameworks do not just restore data; they restore the **state and type** of the object. 

If an attacker can control the serialized byte stream, they can force the application to instantiate arbitrary classes that exist on the application's classpath (often called "gadgets"). By chaining these gadgets together (a "gadget chain"), an attacker can trigger unintended code paths, leading to Remote Code Execution (RCE), file system manipulation, or Denial of Service (DoS) during the deserialization process itself, often before the application has even had a chance to "use" the object.

## Impact

*   **Remote Code Execution (RCE):** The most common and severe impact. Attackers can execute arbitrary system commands in the context of the application process.
*   **Denial of Service (DoS):** An attacker can provide a payload that causes the application to consume excessive CPU or memory (e.g., "Billion Laughs" style attacks or hash collisions) or simply crash the process.
*   **Privilege Escalation:** By injecting objects with modified internal states, an attacker might bypass security checks or assume the identity of an administrative user.
*   **Data Tampering:** Maliciously crafted objects can be used to modify application logic or internal data structures.

## Remediation

The primary defense against deserialization vulnerabilities is to **never deserialize data from untrusted sources**.

### 1. Use Safer Data Formats
Prefer pure data formats like JSON, XML, or Protobuf that do not permit the reconstruction of arbitrary objects or the execution of code logic during parsing. Ensure that your JSON/XML parsers are also configured securely (e.g., disabling DTDs in XML).

### 2. Implementation of Allow-lists (Look-ahead Deserialization)
If your application *must* use native serialization, implement a strict allow-list of classes that are permitted to be deserialized. In Java, this can be achieved using a `ValidatingObjectInputStream` (available in Apache Commons IO) or the `ObjectInputFilter` introduced in Java 9.

### 3. Integrity Checks
If you must send serialized data over the network, sign the data using a Message Authentication Code (MAC) such as HMAC. The application should verify the signature *before* attempting to deserialize the data. This ensures the data has not been tampered with by an attacker.

### 4. Isolate the Deserialization Process
Run the code that performs deserialization in a low-privilege environment (e.g., a sandbox or container) to limit the impact of a potential escape.

## Examples

### Java

#### Vulnerable Code
In this example, the code accepts a Base64 encoded object from a cookie and deserializes it directly. An attacker can craft a payload using a tool like `ysoserial` to achieve RCE.

```java
public void doGet(HttpServletRequest request, HttpServletResponse response) {
    try {
        Cookie[] cookies = request.getCookies();
        for (Cookie cookie : cookies) {
            if (cookie.getName().equals("session_data")) {
                byte[] decoded = Base64.getDecoder().decode(cookie.getValue());
                ByteArrayInputStream bis = new ByteArrayInputStream(decoded);
                ObjectInputStream ois = new ObjectInputStream(bis);
                // VULNERABLE: Direct deserialization of untrusted input
                Object obj = ois.readObject();
            }
        }
    } catch (Exception e) {
        e.printStackTrace();
    }
}
```

#### Secure Code
The secure version uses a whitelist approach (ObjectInputFilter) to ensure only the expected class (`SessionState`) is allowed.

```java
public void doGet(HttpServletRequest request, HttpServletResponse response) {
    try {
        Cookie[] cookies = request.getCookies();
        for (Cookie cookie : cookies) {
            if (cookie.getName().equals("session_data")) {
                byte[] decoded = Base64.getDecoder().decode(cookie.getValue());
                ByteArrayInputStream bis = new ByteArrayInputStream(decoded);
                ObjectInputStream ois = new ObjectInputStream(bis);

                // SECURE: Use an ObjectInputFilter to whitelist classes
                ObjectInputFilter filter = ObjectInputFilter.Config.createFilter("com.example.SessionState;!*");
                ois.setInternalObjectInputFilter(filter);

                Object obj = ois.readObject();
            }
        }
    } catch (Exception e) {
        // Handle rejection
    }
}
```

### Python

#### Vulnerable Code
Python's `pickle` module is notoriously insecure.

```python
import pickle
import base64

def load_user_profile(serialized_data):
    # VULNERABLE: pickle.loads can execute arbitrary code
    data = base64.b64decode(serialized_data)
    return pickle.loads(data)
```

#### Secure Code
Use a standard data-only format like JSON.

```python
import json

def load_user_profile(serialized_data):
    # SECURE: json.loads only handles data, not object instantiation
    return json.loads(serialized_data)
```
