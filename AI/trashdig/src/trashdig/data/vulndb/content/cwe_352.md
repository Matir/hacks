# CWE-352: Cross-Site Request Forgery (CSRF)

### Description
Cross-Site Request Forgery (CSRF), also known as "one-click attack" or "session riding," is a vulnerability where a malicious website, email, or blog causes a user's web browser to perform an unwanted action on a different website to which the user is currently authenticated.

The vulnerability stems from the way web browsers handle session management. Specifically, browsers automatically include ambient credentials—such as session cookies, IP addresses, or Windows Integrated Authentication credentials—with every request made to a domain. If a web application relies solely on these automatically included credentials to authorize a request, an attacker can craft a malicious request (e.g., via a hidden form or a script) that the victim's browser will execute as if it were intentional.

### Impact
The impact of a CSRF attack depends on the capabilities of the target application and the privileges of the victim. If the victim is a standard user, the attacker might perform actions like:
*   Changing the user's password or email address (leading to account takeover).
*   Executing financial transactions or purchasing items.
*   Posting messages or deleting content.

If the victim is an administrative user, a successful CSRF attack can compromise the entire application, allowing the attacker to create new admin accounts, modify site configurations, or extract sensitive data.

### Remediation
To prevent CSRF, applications must ensure that state-changing requests (POST, PUT, DELETE, etc.) require a secret, unique, and unpredictable value that an attacker cannot guess or spoof.

1.  **Anti-CSRF Tokens (Synchronizer Token Pattern):**
    The most common defense is to embed a cryptographically strong, unique token in every state-changing form and validate it on the server. The token should be unique per session or per request.

2.  **SameSite Cookie Attribute:**
    Set the `SameSite` attribute on session cookies to `Lax` or `Strict`. 
    *   `Strict`: Prevents the cookie from being sent on any cross-site request.
    *   `Lax`: Prevents the cookie from being sent on cross-site POST requests but allows it on top-level GET navigations (providing a balance between security and usability).

3.  **Custom Request Headers:**
    For AJAX/API-heavy applications, requiring a custom HTTP header (e.g., `X-Requested-With`) can be effective because browsers restrict the ability of cross-origin scripts to set custom headers using the Same-Origin Policy (SOP) and CORS.

4.  **Verifying Origin/Referer Headers:**
    Check the `Origin` or `Referer` headers to ensure the request originated from a trusted domain. While useful as a secondary check, these headers can sometimes be spoofed or omitted by privacy tools.

### Examples

#### Vulnerable Code (Node.js/Express)
In this example, the application processes a profile update using only session cookies for authentication. An attacker can host a form on a different site that submits to this endpoint.

```javascript
const express = require('express');
const bodyParser = require('body-parser');
const session = require('express-session');
const app = express();

app.use(bodyParser.urlencoded({ extended: false }));
app.use(session({ secret: 'secret-key', resave: false, saveUninitialized: true }));

// VULNERABLE: No CSRF protection
app.post('/update-email', (req, res) => {
    if (req.session.user) {
        const newEmail = req.body.email;
        // Logic to update user email in database
        console.log(`Updating email to ${newEmail} for user ${req.session.user}`);
        res.send('Email updated successfully');
    } else {
        res.status(401).send('Unauthorized');
    }
});
```

#### Secure Code (Node.js/Express with csurf)
Using the `csurf` middleware, the application generates a unique token that must be submitted with the form.

```javascript
const express = require('express');
const bodyParser = require('body-parser');
const session = require('express-session');
const csrf = require('csurf');
const app = express();

app.use(bodyParser.urlencoded({ extended: false }));
app.use(session({ 
    secret: 'secret-key', 
    resave: false, 
    saveUninitialized: true,
    cookie: { sameSite: 'lax' } // Mitigation: SameSite attribute
}));

const csrfProtection = csrf();

// SECURE: CSRF protection enabled
app.get('/form', csrfProtection, (req, res) => {
    // Pass the token to the view
    res.send(`<form action="/update-email" method="POST">
                <input type="hidden" name="_csrf" value="${req.csrfToken()}">
                <input type="text" name="email">
                <button type="submit">Update</button>
              </form>`);
});

app.post('/update-email', csrfProtection, (req, res) => {
    if (req.session.user) {
        // The middleware automatically validates the '_csrf' field against the session
        res.send('Email updated securely');
    } else {
        res.status(401).send('Unauthorized');
    }
});
```
