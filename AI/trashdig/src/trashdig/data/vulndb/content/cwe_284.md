# CWE-284: Improper Access Control

### Description
CWE-284 is a broad category of security vulnerabilities that occur when a software application does not restrict access to a resource (such as a file, a database record, or a specific functionality) from an unauthorized actor. Access control is the process of enforcing the policy that determines which users, processes, or systems are granted access to which resources.

When access control is improperly implemented, it usually falls into two categories:
1.  **Vertical Privilege Escalation:** A lower-privileged user (e.g., a "Guest") is able to access functionality reserved for higher-privileged users (e.g., an "Admin").
2.  **Horizontal Privilege Escalation (IDOR):** A user is able to access resources belonging to another user with the same privilege level (e.g., User A viewing User B's private messages by changing a URL parameter).

### Impact
The impact of improper access control is often severe and can lead to:
*   **Data Breaches:** Unauthorized access to sensitive user information, financial records, or intellectual property.
*   **Data Manipulation:** Unauthorized modification or deletion of critical data.
*   **Account Takeover:** Attackers gaining control over administrative accounts to compromise the entire system.
*   **Compliance Violations:** Failing to meet regulatory requirements like GDPR, HIPAA, or PCI-DSS regarding data privacy and segregation.

### Remediation
To mitigate CWE-284, follow the principle of **Defense in Depth** and **Least Privilege**:

1.  **Deny by Default:** Design the application so that all access is denied unless explicitly permitted.
2.  **Centralize Authorization:** Avoid scattered access control logic. Use a centralized library or middleware to handle authorization consistently across all endpoints.
3.  **Validate on Every Request:** Do not rely on the client to provide authorization status (e.g., hidden fields or local storage flags). Always re-verify the user's session and permissions on the server side for every request.
4.  **Use Indirect Object References:** Instead of exposing internal database IDs (e.g., `user_id=502`), use unpredictable mapping or UUIDs, and always verify that the authenticated user owns the requested resource.
5.  **Role-Based Access Control (RBAC) or Attribute-Based Access Control (ABAC):** Implement structured access models that define clear boundaries between user roles and permissions.

### Examples

#### Vulnerable Code (Python/Flask)
In this example, an attacker can view any user's profile simply by changing the `user_id` in the URL, as the code fails to check if the logged-in user matches the requested ID.

```python
@app.route('/api/user/profile/<user_id>')
@login_required
def get_profile(user_id):
    # VULNERABILITY: No check to ensure current_user.id == user_id
    # A user can access any other user's profile by guessing the ID.
    user_data = db.query("SELECT * FROM users WHERE id = %s", (user_id,))
    return jsonify(user_data)
```

#### Secure Code (Python/Flask)
The secure version ensures that the query is restricted to the identity found in the secure session, or explicitly checks permissions before returning data.

```python
@app.route('/api/user/profile/<user_id>')
@login_required
def get_profile(user_id):
    # FIX: Verify that the authenticated user is requesting their own data
    # or has administrative permissions to view other profiles.
    if str(current_user.id) != str(user_id) and not current_user.is_admin:
        abort(403) # Forbidden
        
    user_data = db.query("SELECT email, username, bio FROM users WHERE id = %s", (user_id,))
    if not user_data:
        abort(404)
        
    return jsonify(user_data)

# ALTERNATIVE FIX (Best Practice): Fetch based on session directly
@app.route('/api/user/my-profile')
@login_required
def get_my_profile():
    # Automatically safe: query uses the session-stored user ID
    user_data = db.query("SELECT * FROM users WHERE id = %s", (current_user.id,))
    return jsonify(user_data)
```
