# CWE-200: Exposure of Sensitive Information to an Unauthorized Actor

## Description
CWE-200 occurs when an application unintentionally reveals sensitive information to an unauthorized actor. This information can range from highly sensitive data like personally identifiable information (PII), credentials, and financial records to technical metadata that assists an attacker in profiling the system. Technical leaks often include stack traces, verbose error messages, database schema details, internal IP addresses, or configuration snippets.

While the exposure may not always lead to immediate compromise, it is frequently a critical component of a multi-stage attack. For example, a leaked stack trace might reveal the exact version of a vulnerable library being used, or a configuration file might reveal internal hostnames used for Server-Side Request Forgery (SSRF).

## Impact
The impact of CWE-200 varies significantly based on the nature of the exposed data:
*   **Privacy Violation:** Exposure of user data (names, emails, SSNs) leading to identity theft and loss of trust.
*   **Account Takeover:** Exposure of session tokens, passwords, or password reset tokens.
*   **System Reconnaissance:** Technical leaks allow attackers to map the internal network architecture, identify software versions, and find hidden entry points.
*   **Compliance Penalties:** Violations of regulations such as GDPR, HIPAA, or PCI-DSS, resulting in heavy fines.

## Remediation
To prevent information exposure, developers should adopt a "deny-by-default" approach to data transmission and error handling.

### General Best Practices:
1.  **Generic Error Messages:** Ensure that end-users only see generic error messages (e.g., "An unexpected error occurred"). Detailed logs should be stored on the server for developers, never sent to the client.
2.  **Data Minimization:** Only return the specific fields required by the front-end. Avoid returning full database objects or using `SELECT *`.
3.  **Disable Debugging in Production:** Ensure all debug flags, development consoles, and verbose logging are disabled in production environments.
4.  **Environment Isolation:** Protect sensitive files like `.env`, `.git`, and `web.config` using server-level restrictions (e.g., `.htaccess` or Nginx `deny all`).
5.  **Output Filtering:** Implement automated scanners to detect sensitive patterns (API keys, credit card numbers) before data is rendered or logged.

## Examples

### Vulnerable Code (Python/Flask)
In this example, the application returns the raw exception message to the user, which could include database connection strings or file paths.

```python
@app.route('/user/<id>')
def get_user(id):
    try:
        user = db.query("SELECT * FROM users WHERE id = " + id)
        return jsonify(user)
    except Exception as e:
        # VULNERABLE: Returns the raw exception details to the client
        return str(e), 500
```

### Secure Code (Python/Flask)
The secure version logs the error internally for debugging but returns a sanitized, generic message to the user.

```python
import logging

@app.route('/user/<id>')
def get_user(id):
    try:
        # Use parameterized queries to prevent SQLi as well
        user = db.execute("SELECT username, bio FROM users WHERE id = ?", (id,)).fetchone()
        if user:
            return jsonify({"username": user[0], "bio": user[1]})
        return "User not found", 404
    except Exception as e:
        # SECURE: Log the detailed error internally
        logging.error(f"Database error: {e}")
        # Return a generic message to the unauthorized actor
        return "An internal server error occurred.", 500
```
