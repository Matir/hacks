### Description

CWE-125: Out-of-bounds Read occurs when a software program reads data past the end, or before the beginning, of the intended buffer. This typically happens during loop iterations, pointer arithmetic, or when an index is calculated based on external input without sufficient validation. In languages like C and C++, which do not perform automatic bounds checking, the system will attempt to read whatever data is present at the calculated memory address, even if that address is not part of the allocated buffer.

This vulnerability is frequently found in code that parses complex binary formats, handles network packets, or performs manual string manipulation.

### Impact

The impact of an out-of-bounds read can vary depending on the memory layout:

1.  **Information Disclosure**: An attacker may be able to read sensitive data from the process's memory that they should not have access to, such as passwords, cryptographic keys, or session tokens.
2.  **Denial of Service (DoS)**: If the application attempts to read from a memory address that is unmapped or protected (e.g., outside the heap or stack limits), the operating system will trigger a segmentation fault, causing the application to crash.
3.  **Bypassing Security Mitigations**: By reading specific memory addresses, an attacker can leak pointers to internal functions or libraries, effectively bypassing Address Space Layout Randomization (ASLR) to facilitate more complex exploits like Return-Oriented Programming (ROP).

### Remediation

To prevent out-of-bounds reads, developers should implement the following strategies:

*   **Input Validation**: Always validate indices, lengths, and offsets against the actual size of the buffer before performing a read operation.
*   **Use Safe APIs**: In C++, prefer using container methods that perform bounds checking, such as `std::vector::at()` instead of the `[]` operator.
*   **Length Tracking**: Ensure that the length of a buffer is always passed alongside the buffer pointer itself to functions that process the data.
*   **Static and Dynamic Analysis**: Utilize static analysis security testing (SAST) tools and dynamic analysis tools like AddressSanitizer (ASan) during the development lifecycle to catch memory errors early.
*   **Modern Languages**: Where possible, use memory-safe languages (e.g., Rust, Java, Python) that automatically manage memory and prevent out-of-bounds access by design.

### Examples

#### Vulnerable Code (C)
In this example, the function reads from an array using a user-provided index without checking if the index is within the valid range of the array.

```c
#include <stdio.h>

int getValueFromArray(int *array, int index) {
    // VULNERABLE: No check to ensure index is within the bounds of the array
    return array[index];
}

int main() {
    int myData[5] = {10, 20, 30, 40, 50};
    // An attacker-controlled index could leak memory
    int value = getValueFromArray(myData, 10); 
    printf("Value: %d\n", value);
    return 0;
}
```

#### Secure Code (C)
In this version, the function explicitly checks the index against the known size of the buffer.

```c
#include <stdio.h>
#include <stdbool.h>

bool getValueFromArraySecure(int *array, size_t size, int index, int *outValue) {
    // SECURE: Validate index against 0 and the actual size of the buffer
    if (index < 0 || (size_t)index >= size) {
        return false; // Handle error appropriately
    }
    *outValue = array[index];
    return true;
}

int main() {
    int myData[5] = {10, 20, 30, 40, 50};
    int value;
    if (getValueFromArraySecure(myData, 5, 10, &value)) {
        printf("Value: %d\n", value);
    } else {
        printf("Error: Index out of bounds\n");
    }
    return 0;
}
```
