# CWE-863: Incorrect Authorization

## Description
CWE-863 occurs when an application incorrectly performs an authorization check when an actor attempts to access a resource or perform an action. While **Authentication** (CWE-287) verifies *who* a user is, **Authorization** (CWE-863) determines *what* that user is allowed to do.

This vulnerability arises when the software does not correctly implement the access control policy, leading to scenarios where a user can perform operations or access data outside of their intended permissions. This often manifests as:
1. **Vertical Privilege Escalation**: A standard user performing administrative tasks.
2. **Horizontal Privilege Escalation (IDOR)**: A user accessing resources belonging to another user of the same privilege level.
3. **Missing Checks**: Failing to verify permissions on specific API endpoints or state transitions.

## Impact
The impact of incorrect authorization is typically severe and includes:
*   **Confidentiality Breach**: Unauthorized access to sensitive data, such as personal information (PII), financial records, or proprietary business data.
*   **Integrity Violation**: Unauthorized modification or deletion of data, leading to data corruption or loss.
*   **Account Takeover**: An attacker may be able to modify other users' credentials or security settings.
*   **Complete System Compromise**: If administrative functions are exposed, an attacker can gain full control over the application environment.

## Remediation
To prevent authorization failures, implement a robust and centralized access control strategy:

1.  **Centralize Authorization**: Avoid scattering authorization logic throughout the codebase. Use a centralized library or middleware to enforce policies.
2.  **Deny by Default**: Configure the application to deny all access unless explicitly permitted.
3.  **Check Every Request**: Perform authorization checks on every single request, not just during the initial login or session creation.
4.  **Validate Ownership**: When accessing a specific resource (e.g., `/api/orders/123`), ensure the authenticated user actually owns or is authorized for that specific resource ID, not just that they have the "User" role.
5.  **Server-Side Enforcement**: Never rely on client-side controls (hidden buttons, UI-only logic) to enforce authorization.

## Examples

### Vulnerable Example (Node.js/Express)
In this example, the application checks if the user is logged in, but it fails to check if the logged-in user actually owns the profile they are trying to update.

```javascript
app.put('/api/user/profile/:id', isAuthenticated, (req, res) => {
    const userId = req.params.id;
    const updatedData = req.body;

    // VULNERABILITY: Any authenticated user can update ANY other user's profile
    // by simply changing the ID in the URL.
    db.users.update({ id: userId }, updatedData)
        .then(() => res.status(200).send("Profile updated"))
        .catch(err => res.status(500).send(err));
});
```

### Secure Example (Node.js/Express)
The secure version validates that the `id` from the URL matches the `id` associated with the authenticated session.

```javascript
app.put('/api/user/profile/:id', isAuthenticated, (req, res) => {
    const resourceId = req.params.id;
    const authenticatedUserId = req.user.id; // From session/JWT
    const userRole = req.user.role;

    // SECURE: Verify ownership or administrative privilege
    if (resourceId !== authenticatedUserId && userRole !== 'ADMIN') {
        return res.status(403).send("Unauthorized: You do not own this resource.");
    }

    db.users.update({ id: resourceId }, req.body)
        .then(() => res.status(200).send("Profile updated"))
        .catch(err => res.status(500).send(err));
});
```

### Vulnerable Example (Java/Spring)
This controller method checks for a valid session but does not restrict access to the sensitive "delete" function based on roles.

```java
@DeleteMapping("/api/admin/delete-user/{id}")
public ResponseEntity<?> deleteUser(@PathVariable Long id) {
    // VULNERABILITY: No check to ensure the user has the 'ADMIN' role.
    // Any authenticated user can call this endpoint.
    userService.deleteUser(id);
    return ResponseEntity.ok().build();
}
```

### Secure Example (Java/Spring)
Using Spring Security annotations to enforce role-based access control (RBAC).

```java
@PreAuthorize("hasRole('ADMIN')")
@DeleteMapping("/api/admin/delete-user/{id}")
public ResponseEntity<?> deleteUser(@PathVariable Long id) {
    // SECURE: Spring Security will intercept the call and ensure 
    // the user has the 'ROLE_ADMIN' before executing the method.
    userService.deleteUser(id);
    return ResponseEntity.ok().build();
}
```
